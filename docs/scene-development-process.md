# Scene 개발 프로세스 (참고용)

> Home Scene(`/home`) 작업에서 실제로 거친 흐름을 일반화한 메모.
> "스크린샷 → 초안 정리 → 대화로 구체화 → 설계 확정 → 구현 계획"의 순서로,
> 다른 Scene을 새로 만들 때 이 순서를 따라가면 된다. (아직 스킬은 아님 — 참고 문서)
> 관련: [scenes.md](scenes.md), 예시 산출물 [superpowers/specs/2026-06-28-home-scene-design.md](superpowers/specs/2026-06-28-home-scene-design.md)

## 핵심 아이디어

**시각 목표(스크린샷)를 기준점으로 삼고, 화면 요소를 하나씩 짚어가며 요구사항을
임시 노트에 누적 → 엣지케이스 리뷰 → 정식 스펙 → 브랜치/커밋 → 구현 계획** 순으로
진행한다. 한 번에 다 정하려 하지 않고, 대화로 한 항목씩 확정한다.

## 단계

### 0. 준비
- 대상 Scene의 **시각 목표 스크린샷**을 받는다 (예: `mytemp/<scene>/XxxScreenshot.png`).
- 쓸 수 있는 **에셋 파일**들이 어디 있는지 확인한다 (예: `mytemp/<scene>/`).
- **임시 작업 노트** 마크다운을 만든다 (`docs/<scene>-notes.md`).
  기본 정보(URL/컴포넌트/흐름상 위치/레이아웃 기준)를 미리 채운다.

### 1. 스크린샷 판독
- 스크린샷을 읽고 화면 구성(상단/중앙/하단, 각 요소)을 글로 정리해 인식을 맞춘다.
- 잘못 읽은 부분은 사용자가 바로잡게 한다.

### 2. 요소별 구체화 (반복)
화면 요소를 **하나씩** 다룬다. 각 요소마다:
1. 사용자가 의도/동작/데이터를 설명한다.
2. Claude가 **에셋 파일을 실제로 열어 요소와 매칭**한다 (이미지 Read).
3. 노트에 기록하고, **초점 잡힌 확인 질문**을 한다 (애매한 점만).
4. 미정 항목(애니메이션/인터랙션/문구 등)은 **Claude가 추천안을 제시**하고
   사용자가 고르게 한다. 선택지를 나열만 하지 말고 추천 + 이유를 준다.
5. 결정되면 노트의 해당 항목을 "확정"으로 갱신한다.

### 3. 기존 코드에 정착(grounding)
- 데이터 출처(세션/스토어/타입), 라우팅, 재사용 가능한 에셋·CSS 패턴을 코드에서 확인한다.
- 새 방식을 만들기 전에 **기존 패턴을 먼저 따른다** (이 프로젝트: 1280×800 CSS 기준,
  `border-image` 9-slice, 순수 로직 분리 + Vitest 등).

### 4. 엣지케이스 리뷰 (브레인스토밍)
- 모든 요소를 훑은 뒤, 빠진 부분을 **한 번에 하나씩** 질문한다.
  자주 나오는 것: 진입/세션 가드, 빈/에러/로딩 상태, 경계값(progress 0/최대),
  레이아웃 기준, 사운드, 데이터 최신화 흐름.

### 5. 설계안 제시 → 승인
- **컴포넌트 분해**(작은 단위 + 순수 로직 분리)와 **에셋 파이프라인**을 제시한다.
- 사용자 승인 후에만 다음으로 넘어간다.

### 6. 정식 스펙 문서화
- `docs/superpowers/specs/YYYY-MM-DD-<scene>-design.md`에 확정 설계를 정리한다.
- 자체 점검(플레이스홀더/일관성/범위/모호성) 후, 사용자 리뷰를 받는다.

### 7. 브랜치 + 커밋
- Scene별 새 브랜치(예: `feat/<scene>-scene`)를 따고 스펙 문서를 커밋한다.
- 구현은 사용자가 시작하라고 할 때까지 대기.

### 8. 구현 계획 → 구현
- writing-plans로 구현 계획을 만든 뒤 구현에 들어간다.

## 이 프로젝트에서 배운 점 (재사용 팁)

- **에셋 투명 여백**: Unity는 자동 트림하지만 웹은 직접 크롭해야 한다.
  Pillow(PIL)로 alpha bbox 트림 + 시트 분할 스크립트화 가능. 결과는 `src/assets/<scene>/`.
- **라벨이 이미지에 박혀** 있는 경우가 있다(행성 이름 등) → 텍스트 별도 렌더 불필요.
- **추천을 적극적으로**: 애니메이션/인터랙션 같은 열린 결정은 추천안+이유를 먼저 낸다.
  (예: "움직임은 현재 선택지에만" 같은 원리로 직관성 확보.)
- **순수 로직 분리**: 상태/매핑 함수는 컴포넌트에서 떼어 `*.logic.ts` + 테스트.
- **임시 노트 → 정식 스펙** 2단계: 대화 중엔 노트에 빠르게 누적, 끝에 깔끔히 스펙으로 옮긴다.
